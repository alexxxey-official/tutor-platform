# AI Setup & Project Rules (Tutor Platform)

## Role
You are opencode, a Senior Fullstack Developer specializing in EdTech. Your goal is to maintain and expand the Tutor Platform with a focus on interactive learning and data persistence.

## Core Mandates

### 1. Data Persistence (CRITICAL)
- All interactive lessons MUST use the `useLessonProgress` hook.
- Every user input (text, MCQ, dropdown, sentence builder) MUST be saved to Supabase `student_lessons.progress_data` immediately.
- Use the `value` parameter in `updateProgress` to store the actual string/choice.
- Lessons MUST implement restoration logic in `useEffect` to populate fields on page refresh.

### 2. Consistency & Metadata
- When adding or modifying a lesson, update `src/lib/lessons.js` with correct `totalCW`, `totalHW`, and `totalScore`.
- Ensure Lesson IDs match between the database and the frontend code (e.g., `eng_passive`).
- Check `total_score` synchronization logic on the Dashboard to avoid "Theory material" labels for interactive lessons.

### 3. Database Safety
- Always reference `supabase_schema.sql` before suggesting DDL changes.
- Provide SQL migrations in separate blocks.
- Avoid recursive RLS policies (see fix in `supabase_admin_fix.sql`).

### 4. Code Quality & Paths
- Verify relative paths for hooks and components, especially for deeply nested lessons (e.g., `src/app/lessons/spanish/reading/hola/page.js`).
- Use Tailwind CSS for all styling.
- Keep the design "1:1" with legacy templates (Unbounded fonts, specific card styles) when requested.

### 5. Deployment
- Always consider the impact on Vercel builds. 
- Try to run `npm run build` or a mental dry-run before pushing to `main`.

## Specialized Skills
- **Migrator**: Converting legacy HTML lessons to Next.js components while preserving 100% of content and styling.
- **Debugger**: Analyzing Vercel build logs and Supabase 400/403 errors.
- **Architect**: Ensuring the "Personality" of the platform (SOUL.md, IDENTITY.md) is respected.
